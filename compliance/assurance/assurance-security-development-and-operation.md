## <a name="how-does-microsoft-implement-secure-development-practices"></a><span data-ttu-id="aa45e-104">Microsoft 如何实施安全开发实践？</span><span class="sxs-lookup"><span data-stu-id="aa45e-104">How does Microsoft implement secure development practices?</span></span>

<span data-ttu-id="aa45e-105">Microsoft 安全开发生命周期 (SDL) 是一个专注于开发和操作安全软件的安全保证流程。</span><span class="sxs-lookup"><span data-stu-id="aa45e-105">Microsoft's Security Development Lifecycle (SDL) is a security assurance process focused on developing and operating secure software.</span></span> <span data-ttu-id="aa45e-106">SDL 为 Microsoft 的开发人员和工程师提供了详细的可衡量安全要求，以减少我们产品和服务中漏洞的数量和严重性。</span><span class="sxs-lookup"><span data-stu-id="aa45e-106">The SDL provides detailed, measurable security requirements for developers and engineers at Microsoft to reduce the number and severity of vulnerabilities in our products and services.</span></span> <span data-ttu-id="aa45e-107">所有 Microsoft 软件开发团队都必须遵循 SDL 要求，并持续更新 SDL 以反映不断变化的威胁环境、行业最佳做法和合规性法规标准。</span><span class="sxs-lookup"><span data-stu-id="aa45e-107">All Microsoft software development teams must follow SDL requirements, and we continuously update the SDL to reflect the changing threat landscape, industry best practices, and regulatory standards for compliance.</span></span>

## <a name="how-does-microsofts-sdl-improve-application-security"></a><span data-ttu-id="aa45e-108">Microsoft 的 SDL 如何提高应用程序安全性？</span><span class="sxs-lookup"><span data-stu-id="aa45e-108">How does Microsoft's SDL improve application security?</span></span>

<span data-ttu-id="aa45e-109">可以从五个开发阶段考虑 Microsoft 的 SDL 流程：要求、设计、实现、验证和发布。</span><span class="sxs-lookup"><span data-stu-id="aa45e-109">The SDL process at Microsoft can be thought of in terms of five phases of development: requirements, design, implementation, verification, and release.</span></span> <span data-ttu-id="aa45e-110">它首先定义软件要求，并注意安全性。</span><span class="sxs-lookup"><span data-stu-id="aa45e-110">It begins by defining software requirements with security in mind.</span></span> <span data-ttu-id="aa45e-111">为此，我们询问有关应用程序必须完成的任务的安全相关问题。</span><span class="sxs-lookup"><span data-stu-id="aa45e-111">To do this, we ask security-relevant questions about what the application must accomplish.</span></span> <span data-ttu-id="aa45e-112">应用程序是否需要收集敏感数据？</span><span class="sxs-lookup"><span data-stu-id="aa45e-112">Does the application need to collect sensitive data?</span></span> <span data-ttu-id="aa45e-113">应用程序是否执行敏感任务或重要任务？</span><span class="sxs-lookup"><span data-stu-id="aa45e-113">Will the application perform sensitive or important tasks?</span></span> <span data-ttu-id="aa45e-114">应用程序是否需要接受来自不受信任的源的输入？</span><span class="sxs-lookup"><span data-stu-id="aa45e-114">Does the application need to accept input from untrusted sources?</span></span>

## <a name="how-does-microsoft-test-source-code-for-common-vulnerabilities"></a><span data-ttu-id="aa45e-118">Microsoft 如何测试常见漏洞的源代码？</span><span class="sxs-lookup"><span data-stu-id="aa45e-118">How does Microsoft test source code for common vulnerabilities?</span></span>

<span data-ttu-id="aa45e-119">为了支持我们的开发人员在代码开发期间和发布后实现安全要求，Microsoft 提供了一套安全开发工具，用于自动检查源代码的安全缺陷和漏洞。</span><span class="sxs-lookup"><span data-stu-id="aa45e-119">To support our developers in implementing security requirements during code development and after release, Microsoft provides a suite of secure development tools to automatically check source code for security flaws and vulnerabilities.</span></span> <span data-ttu-id="aa45e-120">Microsoft 定义并发布供开发人员使用的已批准工具（如编译器和开发环境）的列表，以及 Microsoft 生成管道中自动执行的内置安全检查。</span><span class="sxs-lookup"><span data-stu-id="aa45e-120">Microsoft defines and publishes a list of approved tools for our developers to use, such as compilers and development environments, along with the built-in security checks executed automatically within Microsoft build pipelines.</span></span> <span data-ttu-id="aa45e-121">我们的开发人员使用最新版本的已批准工具来利用新的安全功能。</span><span class="sxs-lookup"><span data-stu-id="aa45e-121">Our developers use the latest versions of approved tools to take advantage of new security features.</span></span>

<span data-ttu-id="aa45e-122">在将代码签入发布分支之前，SDL 需要由单独的审阅者手动检查代码。</span><span class="sxs-lookup"><span data-stu-id="aa45e-122">Before code can be checked into a release branch, the SDL requires manual code review by a separate reviewer.</span></span> <span data-ttu-id="aa45e-123">代码审阅者检查编码错误，并验证代码更改是否满足 SDL 和设计要求，通过功能和安全测试，并可靠地执行。</span><span class="sxs-lookup"><span data-stu-id="aa45e-123">Code reviewers check for coding errors and verify that code changes meet SDL and design requirements, pass functional and security tests, and perform reliably.</span></span> <span data-ttu-id="aa45e-124">他们还会查看关联的文档、配置和依赖项，以确保正确记录代码更改，并且不会导致意外的副作用。</span><span class="sxs-lookup"><span data-stu-id="aa45e-124">They also review associated documentation, configs, and dependencies to ensure code changes are documented appropriately and will not cause unintended side-effects.</span></span> <span data-ttu-id="aa45e-125">如果审阅者在代码审阅期间发现问题，他们可要求提交者重新提交包含建议更改和其他测试的代码。</span><span class="sxs-lookup"><span data-stu-id="aa45e-125">If a reviewer finds problems during code review, they can ask the submitter to resubmit the code with suggested changes and additional testing.</span></span> <span data-ttu-id="aa45e-126">代码审阅者还可以决定完全阻止不符合要求的代码签入。</span><span class="sxs-lookup"><span data-stu-id="aa45e-126">Code reviewers may also decide to block check-in entirely for code that does not meet requirements.</span></span> <span data-ttu-id="aa45e-127">一旦代码被审阅者视为满意，审阅者将提供审批，这是在代码可以继续下一个部署阶段之前所需的。</span><span class="sxs-lookup"><span data-stu-id="aa45e-127">Once the code has been deemed satisfactory by the reviewer, the reviewer provides approval, which is required before code can proceed to the next deployment phase.</span></span>

<span data-ttu-id="aa45e-128">除了安全开发工具和手动代码检查之外，Microsoft 还使用自动化安全工具强制执行 SDL 要求。</span><span class="sxs-lookup"><span data-stu-id="aa45e-128">In addition to secure development tools and manual code review, Microsoft enforces SDL requirements using automated security tooling.</span></span> <span data-ttu-id="aa45e-129">其中许多工具内置于提交管道中，在签入代码和编译新的内部版本时自动分析代码的安全缺陷。</span><span class="sxs-lookup"><span data-stu-id="aa45e-129">Many of these tools are built into the commit pipeline and automatically analyze code for security flaws as it is checked in and as new builds are compiled.</span></span> <span data-ttu-id="aa45e-130">示例包括常见安全缺陷的静态代码分析，以及分析嵌入密码代码的凭据扫描程序。</span><span class="sxs-lookup"><span data-stu-id="aa45e-130">Examples include static code analysis for common security flaws and credential scanners that analyze code for embedded secrets.</span></span> <span data-ttu-id="aa45e-131">必须修复自动安全工具发现的问题，然后新版本才能通过安全审查并获得批准发布。</span><span class="sxs-lookup"><span data-stu-id="aa45e-131">Issues uncovered by automated security tools must be fixed before new builds can pass security review and be approved for release.</span></span>

<span data-ttu-id="aa45e-133">Microsoft 已采用用于管理开放源代码安全性的高级别策略，该策略利用旨在：</span><span class="sxs-lookup"><span data-stu-id="aa45e-133">Microsoft has adopted a high-level strategy for managing open-source security, which leverages tools and workflows designed to:</span></span>

- <span data-ttu-id="aa45e-135">跟踪这些组件的使用位置和方式。</span><span class="sxs-lookup"><span data-stu-id="aa45e-135">Track where and how those components are used.</span></span>
- <span data-ttu-id="aa45e-136">确定这些组件是否具有任何漏洞。</span><span class="sxs-lookup"><span data-stu-id="aa45e-136">Determine whether those components have any vulnerabilities.</span></span>
- <span data-ttu-id="aa45e-137">发现影响这些组件的漏洞时正确响应。</span><span class="sxs-lookup"><span data-stu-id="aa45e-137">Respond properly when vulnerabilities are discovered that affect those components.</span></span>
